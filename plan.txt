### TODO: LLM response handling when tool calls are present
**Problem**: When an LLM responds with both text AND tool calls (e.g., "Hello! Let me check my memory" + memory_read tool), the system was only processing the first tool call. Subsequent iterations would fail with empty responses, and Bob would not respond.

**Root Cause**: MiniMax (and possibly other providers) sometimes returns empty responses after tool execution, causing the loop to exit without a final text response.

**What We Fixed**:
1. Added 529 (overload) to retryable HTTP status codes
2. Gracefully handle empty/None `choices` in `_handle_response()` - return empty text instead of error
3. When LLM returns empty text after tool calls, fall back to last assistant message in conversation history
4. Process ALL tool calls in a response (0..n loop) instead of just the first one
5. Use `RepetitionError` exception for loop control instead of boolean return
6. Added `initial_response_text` fallback when LLM doesn't complete the response

**Remaining Issue**: When an LLM makes multiple tool calls in sequence (e.g., memory_read then write_file), it sometimes considers the task complete after the first tool call and returns empty on the next iteration. The LLM needs to understand compound tasks as a single unit. This is a model behavior issue, not a code bug.

**Example**: "Check memory and create a file" → LLM calls memory_read → gets result → considers task done → doesn't call write_file

### TODO: Debug logging to file
**Problem**: During development, we needed to add debug print statements to trace LLM responses and tool execution. This pollutes stdout and makes it hard to distinguish between actual output and debug info.

**Feature Request**: Implement a logging system that:
1. Writes debug output to a configurable log file (e.g., `harness_debug.log`)
2. Can be enabled/disabled via environment variable or CLI flag
3. Includes timestamp, module name, and log level
4. Doesn't interfere with normal stdout output

**Example usage**:
```bash
HARNESS_DEBUG=1 ./bob.py  # Enables debug logging to harness_debug.log
./bob.py  # Normal mode, no debug output
```

**Implementation ideas**:
- Use Python's `logging` module with a custom handler
- Add `--debug` flag to CLI
- Configurable log file path via `HARNESS_DEBUG_LOG` env var

## Known Issues

### Multiple Tool Calls
When a model returns multiple tool calls in a single response, the system will return a SystemError
with message: `[SYSTEM ERROR: Multiple tool calls detected (N). Tools: tool1, tool2, ... User confirmation required to proceed.]`

This is a safeguard - only the first tool call is executed. If multiple tool calls are needed,
a new execution round must be initiated by the user or controller.

### Structured vs Text-Based Tool Calling

**Current Behavior:**
- `dispatch()` is used for ALL providers (cloud and local) - only structured native `tool_calls` are executed
- Text responses (including JSON in ```json ``` blocks, ```bash blocks, etc.) are NOT parsed as commands
- This cleanly reveals which models properly support structured tool calling

**Test Results:**
- **gemma4:e4b (ollama)**: ✅ Uses structured `tool_calls` correctly
- **qwen2.5-coder:14b (ollama)**: ❌ Puts JSON in text format, not structured tool_calls
- **MiniMax-M2.7 (cloud)**: ✅ Uses structured `tool_calls` correctly

**Future Enhancement:**
Create granular text-parsing flags in provider config to enable specific text-based parsing:

```python
# Example provider config attributes
attributes: {
    "text_parse_json_codeblock": true,   # Parse ```json ... ``` 
    "text_parse_json_raw": true,          # Parse bare {...} JSON
    "text_parse_bash": true,              # Parse ```bash ... ```
    "text_parse_xml": true,               # Parse <tool>...</tool>
    "text_parse_colon_xml": true,         # Parse <tool>:{...}</tool>
}
```

Each flag is additive - models can have any combination enabled.
Default (all false): only structured native `tool_calls` are executed.

This allows fine-grained control over which text formats to interpret as tool calls,
without forcing all text parsing on or off for a given provider.

### Provider Recommendations File

Created `provider-recommendations.json` with recommended attributes for each provider:

- **Cloud models** (MiniMax, OpenRouter): `enable_small_model_guidance: false`
- **Ollama models that work with structured tool_calls** (gemma4): no text parsing needed
- **Ollama models that don't use structured tool_calls** (qwen2.5-coder variants): 
  enable `text_parse_json_codeblock` and `text_parse_json_raw`
- **Smaller models** (qwen2.5-coder:3b, qwen2.5-coder:1.5b): 
  enable `enable_small_model_guidance: true` for extra guidance

This file provides baseline recommendations that can be applied via `update_provider()`
or by manually merging into `providers.json`.
