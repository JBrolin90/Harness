1. Regex-based vs. Native Tool Calling
Bob's way: You send a massive block of text, the model predicts text that looks like !WRITE, and your Python script uses a Regex to "scrape" that out. If the model adds a single space before the ! or forgets a newline, the Regex might fail, the harness ignores the command, and the model gets stuck in a loop wondering why the file wasn't written.
Pi's way (Native): The API actually has a separate "tools" parameter. When the model wants to write a file, it doesn't just type text; it sends a specific "call" signal that the API identifies before the response even reaches the user. 2. Serial vs. Parallel Execution
This is likely the biggest "Total Time" culprit.

Bob's way (Serial): In your controller.py, you find the first match and execute it. If Bob needs to !READ three files to understand how to write test_controller.py, he has to do three separate round-trips to the cloud.
Pi's way (Parallel): Professional agents can emit multiple tool calls in one message. Pi sees "I need to read these 3 files," executes all three simultaneously, and feeds the results back in one go. You save minutes of network latency and "prefill" time.
3. Typed Schemas
"Typed schemas" mean the model is given a strict JSON structure it must follow. This prevents the "Invalid format" errors that cause Bob to retry. Because Bob is just typing raw text, he has more "creative freedom," which is exactly what leads to parsing errors and retries.

Dual Execution Strategy: Large vs Small Models
===============================================

Background
----------
Smaller local models (e.g., qwen2.5-coder:7b) struggle with:
- Following complex multi-tool regex patterns
- Maintaining context across parallel execution
- Precise delimiter parsing in WRITE/EDIT blocks

Larger cloud models (e.g., MiniMax-M2) can handle:
- Parallel tool execution
- Complex delimiter parsing
- State management across multiple rounds

Proposed Solution
-----------------
Implement two distinct execution modes in controller.py:

1. SERIAL_MODE (for small/local models)
   - One tool per response cycle
   - Simpler regex: basic !TOOL pattern matching only
   - No delimiter blocks - use direct content after command
   - More explicit instruction in system prompt
   - Human-in-the-loop gate more permissive for simple commands

2. PARALLEL_MODE (for large/cloud models)
   - Multiple tools per response cycle (re.finditer)
   - Full delimiter parsing (<<<WRITE_BLOCK>>>, etc.)
   - History compaction to prevent context debt
   - Batch result aggregation

Implementation Approach
----------------------
Add a "execution_mode" attribute to ProviderConfig in provider.py:

    @dataclass
    class ProviderConfig:
        ...
        execution_mode: str = "serial"  # "serial" or "parallel"

Controller logic:

    if current_provider.execution_mode == "parallel":
        # Use re.finditer + batch execution + history compaction
    else:
        # Use simple first-match regex + single tool execution

System prompt variations:
- Serial: "Output ONE tool per response. Format: !TOOL /path\n[content on next line]"
- Parallel: Full delimiter-based parsing with <<<BLOCK>>> markers
